# AdBudgetSentry 项目开发说明

本文档说明当前项目的目录结构、关键模块、数据表和后续开发建议，方便继续迭代。

## 1. 项目目标

当前项目目标不是“把所有逻辑都塞进油猴脚本”，而是拆成三层：

- 油猴脚本：页面采集代理
- 本地后端：持久化、分析、告警记录、后台页
- 后台管理页：观察实例健康和告警结果

## 3. 当前数据流

### 采样链路

1. 油猴脚本抓取页面上的“今日总消耗”
2. 脚本从本地短期历史里找到窗口基线
3. 计算当前值和基线差值
4. 上报到后端 `POST /ingest`
5. 后端写入 `capture_events`
6. 后端做规则分析
7. 命中条件时后端调用 DeepSeek 或本地模型

### 告警链路

1. 油猴脚本判断是否超过前端阈值
2. 超过时调用 PushPlus
3. PushPlus 返回后，把结果写入 `POST /alert-record`
4. 后台页显示最近告警记录

### 健康链路

1. 油猴脚本周期发送 `POST /heartbeat`
2. 后端更新 `script_instances`
3. 后台页按心跳、最近采集、错误数聚合状态

## 4. 数据库表说明

当前数据库为 SQLite，默认路径：

```text
E:\Code\AdBudgetSentry\data\app.db
```

### `script_instances`

按脚本实例聚合后的当前状态表。

主要字段：

- `instance_id`
- `account_id`
- `account_name`
- `last_heartbeat_at`
- `last_capture_at`
- `last_capture_status`
- `consecutive_error_count`
- `health_status`

### `script_heartbeats`

心跳明细表。

用于追踪：

- 实例是否还在线
- 页面是否可见
- 最近一次采集状态

### `capture_events`

采样明细表。

用于保存：

- 当前总消耗
- 窗口增量
- 基线值
- 原始上下文

### `error_reports`

脚本错误明细表。

### `analysis_summaries`

规则分析和模型分析结果表。

### `alert_records`

PushPlus 告警发送回执表。

用于后台展示：

- 最近是否真的发出
- 哪个实例触发
- 标题和摘要
- 成功、失败、跳过

## 5. 当前前端脚本的关键设计

### 5.1 为什么不直接读数据库

当前脚本运行在浏览器环境，不能把“直接读取本机 SQLite”当成稳定方案。更合理的职责边界是：

- 前端负责当前页采样和本地短期快照
- 后端负责长期历史和中心化分析

### 5.2 十分钟前对比值怎么来

当前脚本通过 `GM_setValue("spend_history")` 维护本地短期历史。

这份历史不是页面返回的“十分钟前原始记录”，而是脚本自己每次采样留下来的快照。比较逻辑是：

- 当前时刻抓到的“今日总消耗”
- 减去窗口开始点附近的历史快照值

这是目前最现实的方案，因为页面本身通常只提供“当前累计值”，不直接给“十分钟前值”。

### 5.3 为什么之前会出现报警后归零

之前脚本有两个问题：

1. 只在启动时采一次数据，后续主要依赖整页刷新
2. 刷新后如果 3 秒内表格还没稳定，就抓不到值

结果就是：

- 面板重建时默认值是 `0`
- 新页面如果没抓到有效数据，看起来就像“报警后清空了”

当前已经做了修复：

- 恢复上次成功采样值
- 采样从单次改成持续轮询
- 抓取增加重试
- 基线选择更合理

## 6. 后续建议

### 近期建议

- 继续观察这版脚本是否还会出现“刷新后归零”
- 如果仍偶发抓不到数据，继续增强选择器和重试策略
- 在后台页增加“最近采样值趋势图”

### 中期建议

- 把告警触发逻辑逐步后移到后端
- 后端增加实例详情页
- 增加按账号筛选

### 长期建议

- 如果要支持更多实例和多人访问，数据库升级到 PostgreSQL
- 如果要统一窗口对比值计算，新增后端历史窗口接口，由后端返回基线和差值

## 7. 当前结论

当前系统里最值得保持的原则是：

- 前端不要直接读数据库
- 前端负责采集和短时快照
- 后端负责中心化历史、分析和展示

这条边界能保证现在先跑通，后面也能平滑升级。

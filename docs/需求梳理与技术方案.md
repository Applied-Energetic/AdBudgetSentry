# 需求梳理与技术方案

## 1. 项目背景

在协助某电视品牌进行快手直播投流过程中，发现磁力金牛平台在高客单价、目标成本设置偏高的场景下，可能倾向于引入低质量流量，形成如下风险链路：

1. 平台算法优先追求表层 ROI 或低转化成本。
2. 系统引入疑似激励用户、秒退订单、刷单型流量。
3. 广告账户在 30 分钟内快速消耗数百到上千元。
4. 实际有效成交未同步增长，退款和无效转化风险抬升。
5. 广告主缺少实时预警，只能依赖人工盯盘。

项目目标是用低成本、可快速部署的方式，构建一套“监控发现异常 -> 智能分析归因 -> 及时报警触达”的闭环。

## 2. 业务问题识别

### 2.1 关键痛点

- 平台原生缺少分钟级实时预警。
- 页面数据偏前端展示，缺少可直接复用的开放接口。
- 消耗异常不一定代表坏事，存在“正常爆量”和“低质流量引入”混淆。
- 广告主对是否立刻停投、调预算、切计划缺少统一判断依据。

### 2.2 典型异常模式

- `异常退款型`: 转化表面存在，但退款比例快速升高。
- `场观突增型`: 流量突然放大，但成交或停留质量未同步改善。
- `无转化狂飙型`: 消耗短时快速上升，订单数持续为零。
- `停投失速型`: 预算未耗尽，但消耗速率突然掉到接近零。

## 3. 产品目标

### 3.1 核心目标

- 在不依赖正式后端的情况下实现分钟级监控。
- 将模糊痛点转化为可量化指标。
- 提供“是否需要暂停计划”的辅助判断。
- 形成可以开箱即用的报警工具。

### 3.2 用户价值

- 降低人工盯盘成本。
- 缩短异常发现时间。
- 提高对低质流量的识别能力。
- 为后续和平台沟通赔付争议保留时间序列证据。

## 4. 指标体系

建议监控的基础指标：

- 今日总消耗
- 分钟级消耗增量
- 分钟级消耗速率
- 场观变化
- 成交单量
- 退款单量 / 退款比例
- ROI / 直接 ROI
- 转化成本
- 无成交持续时长

衍生指标：

- `rolling_spend_delta`: 滑动窗口消耗增量
- `hour_bucket_zscore`: 同小时段动态 Z-Score
- `conversion_efficiency_shift`: 转化效率偏移
- `refund_pressure`: 退款压力分数
- `low_quality_traffic_score`: 低质流量嫌疑分数

## 5. 功能拆解

### 5.1 已有能力

- Tampermonkey 注入页面
- DOM 抓取财务 / 推广页数据
- 滑动窗口比较消耗变化
- PushPlus 邮件与微信通道报警

### 5.2 本轮扩展

- 新增智能分析网关
- 支持 `local` 与 `deepseek` 两种分析路线切换
- 统一输出自然语言异常归因
- 为后续接入退款、场观、成交等多维数据保留协议

## 6. 技术方案

### 6.1 前端采集层

- 技术：`Tampermonkey + JavaScript + jQuery`
- 职责：
  - 抓取财务页“今日总消耗”
  - 维护本地时间序列历史
  - 进行基础阈值监控
  - 在触发异常时调用本地分析网关

### 6.2 智能分析层

- 技术：`Python`
- 职责：
  - 接收前端上报的时间序列与事件上下文
  - 计算动态 Z-Score 和基础异常特征
  - 选择分析提供方：
    - `local`: 本地 OpenAI 兼容模型服务，如 Ollama / LM Studio / vLLM
    - `deepseek`: 转发到 DeepSeek API
  - 输出归因摘要、风险等级、操作建议

### 6.3 触达层

- 技术：`PushPlus API + QQ 邮箱消息通道`
- 输出：
  - 短报警标题
  - 结构化风险摘要
  - AI 归因结论

## 7. 两条智能分析路线

### 7.1 路线 A：本地大模型

适合：

- 数据敏感，不希望把投流数据发到外部 API
- 希望低边际成本长时间运行
- 可以接受首次部署复杂度更高

优点：

- 数据留在本机
- 成本可控
- 可进一步做轻量微调 / LoRA

缺点：

- 部署复杂
- 受本地显卡显存与吞吐限制
- 输出稳定性依赖模型与 prompt 调参

### 7.2 路线 B：DeepSeek API

适合：

- 先追求上线速度与分析质量
- 希望减少本地部署维护成本
- 可以接受外部 API 调用成本与数据出域

优点：

- 集成快
- 推理质量更稳定
- 便于快速验证价值

缺点：

- 存在 API 成本
- 网络依赖更强
- 需要处理数据脱敏与密钥管理

## 8. 推荐落地顺序

1. 先用当前 Tampermonkey 监控链路跑通分钟级采集。
2. 上线 Python 分析网关，默认先接 `deepseek` 验证智能分析价值。
3. 并行评估本地 `Qwen2.5` 推理质量。
4. 当本地模型达到可用阈值后，再切到 `local` 模式降低长期成本。
5. 若后续样本积累足够，再评估 3B 级模型 LoRA 微调。
